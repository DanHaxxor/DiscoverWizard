DiscoverWizard Anatomy

How the wizard turns a user's description of their business into a ranked recommendation — the nine questions, the role of the language model, the four-layer pipeline, and where the guardrails are.

The problem it's trying to fix is an old one for Zoho: customers buy the platform, open a catalog of forty-five apps, try to stand up eight of them at once, and churn before a single core workflow is running. The usual answer is a shallow "pick one app" quiz. DiscoverWizard is an attempt at something else — a wizard that asks about the business first and the software last.

The nine questions

Phase I collects nine answers across three mini-phases and produces a platform-agnostic diagnosis. Zoho is never named in Phase I — by design. The user should see their problem clearly before any brand enters the conversation.

Demographics

  • D1, D2Industry and sub-industry.
  • D3How many people work at the company.
  • D4What best describes the user's role. This one is load-bearing: role decides whether the final report is written to act on (decision-makers) or to bring to leadership (everyone else).

Pain points

  • PP1Where the operation is breaking down most — sales, service, ops, or marketing. This branch selector rewrites PP2 and PP3 to use the right language for that area.
  • PP2How the user is handling this today.
  • PP3What they think is really driving it.
  • PP4How they'd describe the challenge in their own words. Free text. This is the field that does the heavy lifting, and it's the only field the language model ever reads.

Impact and urgency

  • IU1How the problem is showing up in the business — lost revenue, wasted time, churn, frustration, general drag. Multi-select.
  • IU2How pressured they feel to solve it, on a four-step scale from planning ahead to on fire right now.

Why PP4 matters

Dropdowns capture facts. PP4 captures how the user frames their own problem — the verbs they use, what they blame, how urgent it sounds in their own voice. That's where nuance lives, and it's where the language model earns its keep.

The model reads PP4 and returns three tags, each drawn from a fixed vocabulary. root_cause_type lands on one of process, people, technology, or data. problem_pattern picks from seven shapes of dysfunction — firefighting, leakage, bottleneck, visibility, adoption_failure, missing_ownership, communication_breakdown. urgency_signal is a simple low / medium / high.

Alongside the tags, the model returns a one- or two-sentence interpretation in plain English, and the specific quotes from the user's text that justify each tag. The controlled vocabulary is the point: the model isn't inventing labels, it's picking from a menu. That's what makes its output stable enough to feed into a rule engine downstream.

Where the AI stops

The single most important architectural decision in this app: the language model parses, it does not recommend. Every recommendation a user sees comes from code, not from the model.

The LLM labels the problem. The rule engine picks the answer.

What the LLM does

  • Reads PP4 and its context.
  • Chooses one value per tag from the fixed vocabulary.
  • Writes a short interpretation in plain English.
  • Surfaces the quote that justifies each tag.
  • Writes the diagnostic narrative — without brand names.

What it never does

  • Pick which Zoho products to recommend.
  • Decide rollout order.
  • Score or rank solution patterns.
  • Name a brand in Phase I output.
  • Proceed if the output doesn't match the schema.

How a recommendation is actually made

Once the model has labeled the problem, a deterministic pipeline turns those labels — plus the structured answers — into a ranked recommendation. Four layers, each one auditable:

  1. Input

    The nine structured answers plus the free-text PP4.

  2. Signals

    Structured answers convert to internal signals — things like crm_needed at weight 1.0 or workflow_automation at 0.8. Role, team size, and customer type act as modifiers that boost or dampen specific signals.

  3. Patterns

    Four solution patterns compete. Each defines required signals (a gate), anti-signals (a veto), and supporting signals (a confidence boost). The winner is whichever pattern clears the gate with the highest combined score.

  4. Bundles

    The winning pattern maps to a concrete set of Zoho apps, ordered by rollout priority, with an integration-aware "why this product" sentence attached to each.

The four solutions in this build

The scoring engine currently knows about four solutions, one per breakdown area. That's down from fifteen in an earlier iteration — deliberately, because a smaller set makes the logic fast to tune and easy to audit while we stabilize the core diagnostic.

  • Sales Pipeline and follow-up automation Zoho CRM · SalesIQ · Campaigns
  • Customer service Unified inbox Zoho Desk · SalesIQ · CRM
  • Internal ops Workflow automation Zoho Creator · Cliq · Projects
  • Marketing Lead attribution Zoho Marketing Automation · Campaigns · Social

Calibration built in

Every preset on the debug screen ships with an expected set of tags — what we believe the model should produce for that input. After each run, the accuracy tracker records which tags matched and which didn't, and keeps a rolling history in local storage.

The tracker turns the debug screen into a lightweight evaluation harness. If problem_pattern is landing at 60 percent while urgency_signal is at 95, that's a specific signal about which part of the prompt needs tuning. Bad tags cascade — the rule engine will happily pick the wrong pattern based on the wrong label — so tracking accuracy per tag matters more than tracking accuracy as a single number.

What this build isn't doing yet

Phase II — the richer Zoho-specific conversation with case studies, playbooks, and multi-app orchestration — isn't wired. The LLM call has no retry and no fallback model; a failed call surfaces the error to the UI. The accuracy tracker shows drift but doesn't feed bad runs back into prompt tuning automatically. And the consumer-facing wizard itself has been stripped: this build is the debug harness only, while the core diagnostic stabilizes.
