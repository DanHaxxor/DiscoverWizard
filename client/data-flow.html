DiscoverWizard — Data Flow

End-to-end trace of what happens when a user clicks Run diagnostic on the debug screen. Phase I only (platform-agnostic diagnosis wired through a 4-pattern rule engine).

1. Actors

LayerFileRole
Client UI client/index.html Debug screen — structured dropdowns, free text, output panels, session accuracy tracker.
Client — scoring client/scoring-engine.js Pure function: score(answers, knowledgeModel, supplementarySignals) → ranked patterns + bundles.
Client — rendering client/reasoning-card.js Pure DOM: render(answers, scoring, debug, knowledgeModel, expected) populates output IDs.
Client — knowledge client/knowledge-model.json 4 solution patterns, 18 signals, 10 products, modifiers.
Server functions/discowizard_function/index.js Catalyst serverless function: auth → LLM call → parse → log → return.
LLM External Zoho QuickML endpoint Free text → structured tags + diagnostic narrative.
Persistence (server) Catalyst Data Store WizardSessions Fire-and-forget row per run.
Persistence (client) localStorage.discoverwizard_debug_accuracy_v1 Session accuracy tracker, ≤200 runs rolling.

4. Signal flow (the four-layer model)

  ┌───────────────────┐
  │ 1. User input     │  structured dropdowns + free text
  └─────────┬─────────┘
            ▼
  ┌───────────────────┐
  │ 2. Signals        │  ← buildSupplementarySignals(answers) — structured → weighted tags
  │                   │  ← LLM parsedSignals — free text → 3 high-level tags
  │                   │  ← modifiers — role × size × b2b multipliers
  └─────────┬─────────┘
            ▼
  ┌───────────────────┐
  │ 3. Patterns       │  4 patterns compete: required_signals gate, anti_signals veto,
  │                   │  supporting_signals + profile + cohesion = confidence
  └─────────┬─────────┘
            ▼
  ┌───────────────────┐
  │ 4. Bundles        │  primary/supporting Zoho products with integration-aware "why"
  └───────────────────┘
AI boundary: the LLM is a parsing layer only. It does not pick patterns or products. All ranking logic lives in scoring-engine.js and is fully auditable.

5. What's not in this diagram (future work)

  • Phase II: mapping the Phase I diagnostic to the 4 patterns currently happens via breakdownArea alone. A richer LLM-tags → pattern-selection layer is open.
  • Feedback loop: the accuracy tracker is client-only. A round-trip that feeds bad runs back into prompt tuning (few-shot examples, tag definitions) is not wired.
  • Session durability: only LLM runs are logged server-side (WizardSessions). Scoring output, user corrections, and tracker state are not persisted centrally.
  • Retry / fallback: the spec calls for hardening the LLM path (retry, fallback model). Currently a single attempt, errors surfaced to the UI.
